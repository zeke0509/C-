# 介绍一下TCP/IP模型和OSI模型的区别
- OSI模型是用于**计算机或通信系统间互联的标准体系**，它将计算机网络通信划分为7个层级，从下至上分别是**物理层、数据链路层、网络层、传输层、会话层、表示层和应用层**
- 而在实际网络通信中，`TCP/IP`模型的4层模型更为实用，从下至上分别是**网络接口层、网络层、传输层和应用层**
  首先，网络接口层对应OSI模型的物理层和数据链路层。它负责物理传输媒介的数据传输、封装和解析帧、错误检测及纠正和MAC地址管理。
  网络层对应于OSI模型的网络层，它负责路由和寻址。路由是选择最佳路径将数据包从源主机传输到目的主机。寻址使用I**P协议来标识主机和网络**，并进行逻辑地址寻址
  传输层对应于OSI模型的传输层，负责端到端的数据传输。主要的传输层协议有TCP和UDP。TCP提供可靠的数据传输，确保数据的正确性和完整性，而UDP是无连接的，适用于不要求可靠性的传输，如实时音频和视频流
  应用层对应于OSI模型的会话层、表示层和应用层。它负责提供直接与用户应用程序交互的接口，为网络上的各种应用程序提供服务，如电子邮件、网页浏览、文件传输等

# 从输入 URL 到页面展示到底发生了什么？
### 总结：
`URL地址输入 -> DNS解析 -> 建立TCP连接 -> 发送Http/https请求 -> 服务器返回数据 -> 浏览器解析并渲染页面 -> 断开TCP连接`
## 回答
URL是统一资源定位符，也就是在网页中输入的网址。
1. 第一步，浏览器**解析**网址是否合法和完整
2. 第二步，DNS域名解析，以获取请求域名的**IP地址**。DNS解析依次从**本地浏览器缓存 -> 本地host文件 -> 路由器缓存 -> DNS服务器 -> 根DNS服务器**的顺序查询域名对应的IP
3. 第三步，浏览器与查询到的服务器IP通过**TCP三次握手**建立TCP连接
4. 第四步，客户端发送HTTP请求。浏览器构建**请求行、请求头和该域名相关的Cookies等数据（加入请求头中）**的打包，向服务器构建请求信息。（如果是HTTPS的话，还涉及HTTPS的加密流程）
5. 第五步，服务器处理HTTP请求，对数据包进行解析，根据请求生成响应数据并返回HTTP资源
6. 第六步，浏览器解析响应。
   若响应头状态码为301、302，会重定向到新地址；
   若响应数据类型是字节流类型，会将请求提交给下载管理器；
   若是HTML类型，会进入浏览器渲染流程。浏览器解析HTML文件，创建DOM树，解析CSS进行样式计算，然后将CSS和DOM合并，构建渲染树；最后布局和绘制渲染树，完成页面展示
7. 第七步，标签页关闭，TCP通过四次挥手断开浏览器与服务器IP的TCP连接

# 说一下select，poll和epoll区别
`select、poll、epoll`都是系统实现**IO多路复用**的机制，多路复用指允许单个线程监视多个**文件描述符**的就绪状态
`select`是最早的IO多路复用机制，内核通过**轮询**的方式检查`fd`的就绪状态，**时间复杂度为`O(n)`**，每次调用要把这个`fd`集合从用户态拷贝到内核态。返回值是`int`类型，表示就绪事件`fd`的数量，**底层数据结构是`bitmap`导致文件描述符上限为1024**
`poll`和`select`最大的不同是把**底层数据结构从`bitmap`改成`pollfd`结构数组**，这样就没有了文件描述符上限，时间复杂度仍然是`O(n)`
`epoll`是`Linux`特有的**高性能IO多路复用机制**，底层数据结构为**红黑树**存储，并维护一个**就绪链表**直接返回已就绪的描述符，时间复杂度是`O(1)`，同时也没有文件描述符的上限限制。`epoll`的实现过程，首先使用`epoll_create()`创建`epoll`实例，然后通过`epoll_ctl`注册/管理`fd`到内核的红黑树中（添加/删除`fd`的时间复杂度为`O(logn）`，最后调用`epoll_wait()`等待事件，只返回已就绪的就绪链表。同时支持**水平触发**和**边沿触发**
## 追问
### Q1.为什么epoll比select/poll更高效？

epoll使用**事件驱动（IO就绪时通知）而非轮询**，只关注活跃连接

epoll使用红黑树管理描述符，查找效率高

epoll_wait直接返回就绪列表，无需遍历所有连接

内核与用户空间共享内存，减少数据拷贝

### Q2.epoll的LT和ET模式有什么区别？如何选择？

LT(水平触发): 只要fd就绪就会不断通知

ET(边缘触发): 只在fd状态变化时通知一次

ET效率更高但编程更复杂，需要一次处理完所有数据

默认使用LT，高性能场景使用ET

### Q3.epoll是如何实现高性能的？

使用红黑树存储监控的fd，插入删除效率高

就绪列表直接返回活跃fd，无需遍历

使用mmap共享内存减少内核与用户空间数据拷贝

回调机制避免轮询

### Q4.select的1024限制可以修改吗？

可以修改FD_SETSIZE宏并重新编译程序

但不推荐，因为内核可能有自己的限制

需要大文件描述符数量时应改用poll/epoll

# HTTP请求报文和响应报文是怎样的，有哪些常见的字段？
`HTTP`报文分为**请求报文**和**响应报文**
请求报文由请求行、请求头、空行和请求体构成。
请求行包括
- 方法：指定要执行的操作，如`GET、POST`
- ==资源路径：请求资源的`url`==
- `HTTP`版本：使用的`HTTP`协议版本，如`HTTP/1.1 或 HTTP/2.0`
请求头：以**键值对**的形式，告诉服务器，**客户端的额外信息**。常见的字段有：
- `Host`：域名
- `Accept`：客户端能处理的媒体类型
- `Cotent-Length`：请求体的长度
- `Cookies`：存储在客户端的`Cookie`数据
请求体：主要在`POST/PUT`请求中使用，存放客户端要提交给服务端的数据，如表单、密码

`HTTP`响应报文同样分为三部分：状态行、响应头、空行、响应体
状态行包含`HTTP`版本、状态码和==状态消息==
响应头也是以键值对的形式提供额外信息，用于告知客户端，有关**响应的详细信息**，主要字段有：
- Content-Type：指定响应主体的媒体类型。
- Content-Length：指定响应主体的长度（字节数）。
- **`Expires`**: 响应的过期时间，之后内容被认为是过时的。
- **`Last-Modified`**： 资源最后被修改的日期和时间。
- - Location：在重定向时指定新的资源位置
响应体：是服务器实际传输的数据，可以是文本、`HTML`页面、图片、视频、空等

## GET请求和POST请求的区别
1. **用途**：GET请求通常用于获取数据，POST请求用于提交数据。
2. **数据传输**：GET请求将参数附加在URL之后，POST请求将数据放在请求体中。
3. **安全性**：GET请求由于参数暴露在URL中，安全性较低；POST请求参数不会暴露在URL中，相对更安全。
4. **数据大小**：GET请求受到URL长度限制，数据量有限；POST请求理论上没有大小限制。
5. **幂等性**：GET请求是幂等的，即多次执行相同的GET请求，资源的状态不会改变；POST请求不是幂等的，因为每次提交都可能改变资源状态。
6. **缓存**：GET请求可以被缓存，POST请求默认不会被缓存。

# 什么是强缓存和协商缓存
强缓存和协商缓存是HTTP缓存机制的两种实现方法，它们用于减少服务器的负担和提高网页加载速度。
- 强缓存是指：浏览器不访问服务器，直接用本地缓存。
  强缓存通过首次响应报文中的`expires`和`cache-control`两个字段进行判断。`cache-control`的优先级要高于`expires`。`cache-control`是一个相对的过期时间，而`expires`是绝对的过期时间
- 协商缓存是：当强缓存失效，浏览器会发送`http`请求到服务器，服务器通过两种方式确定响应报文中的状态码是否为304，若为304，就不发送资源内容，客户端可以直接用浏览器缓存。第一种是基于时间的方式，通过比较请求头中的If-Modified-Since`和响应头的Last-Modified`关键字实现；第二种是基于哈希码的方式，通过比较请求头中的If-None-Match 和响应头中的`etag`字段查看资源是否修改。第二种`etag`的方式更加可靠

# `HTTP1.0`、`HTTP1.1`、`HTTP2`、`HTTP3`的区别

(1) `HTTP1.0`
- `HTTP1.0`默认**短连接**，每次请求都需要建立一个`TCP`连接，每次建立连接和断开都需要经过3次握手，4次挥手，造成网络资源浪费。
- 每次请求对应一个响应，且必须等待前一个响应完整返回，才能处理下一个请求，所以容易发生**队头阻塞**
- 若客户端只要某个对象的一部分，服务器也会将整个资源对象发送过来，且不支持断点续传，造成**带宽资源浪费**
(2) `HTTP1.1`
- 默认使用`TCP`长连接，允许在一个`TCP`连接上发送多个`HTTP`请求和响应，减少了连接建立和关闭的开销
- `HTTP1.1`支持**管道化，允许客户端在第一个请求响应到达之前发送第二个请求，这可以减少等待时间，提高效率。
- 但是`HTTP1.1`要求响应也是按顺序返回，所以也容易发生队头阻塞
- `HTTP1.1`**引入了`Etag/If-None-Match`的缓存头来控制缓存策略**
- `HTTP1.1`引入了一些**新的`HTTP`状态码**，如100`Continue`，用于增强对中间响应的处理
- `HTTP1.1`**引入了`Host`头**，允许客户端指定请求的主机名，这使得同一台服务器上托管多个域名成为可能。
- `HTTP1.1`请求头引入了`range`头域，允许只请求资源的某个部分，返回码是206
(3)`HTTP2.0`
- 采用二进制格式传输数据，而非`HTTP1.1`的文本格式，使得解析更高效
- 支持**多路复用**，允许单个`TCP`连接上并行交错发送多个请求和响应；但是`TCP`端到端的队头阻塞仍然存在，如`TCP`传输过程中**出现丢包，会阻塞重传**
- **引入`HPACK`压缩算法**，对请求和响应的**头部信息进行压缩**，减少了冗余头部信息的传输，提高了传输效率
- 允许服务器**主动推送**资源给客户端，而不需要客户端明确请求，减少页面加载时间
- 允许客户端**为请求设置优先级**，并表达请求之间的依赖关系，**资源加载更有序**
(4) `HTTP3.0`
- `HTTP3.0`基于`QUIC`协议，使用`UDP`协议传输数据，一个连接上的多个`stream`之间没有依赖，如果一个`stream`丢了一个`UDP`包，只会阻塞当前的流，而不会影响后面的`stream`，**不存在队头阻塞**
- **`零RRT`建立连接**，`HTTP3.0`的`QUIC`协议会同时完成**传输层+`TLS`握手**减少了连接延迟，加快了页面加载速度
- `QUIC`允许**网络切换时**，将连接迁移到新的IP地址，从而减少链接的中断时间（底层实现是：用`Connection ID`替换了(IP,端口)四元组）

# HTTPS和HTTP有哪些区别
HTTP是**明文传输**，容易受到攻击；`HTTPS`在`HTTP`的基础上增加了`SSL/TLS`协议作为加密层，确保数据传输的安全性。
`HTTPS`在`TCP`三次握手之后，还要进行`SSL/TLS`握手。
`HTTPS`协议需要向**CA申请数字证书**，来保证服务器的身份是可信的
# HTTPS的工作原理(HTTPS建立连接的过程）
`HTTPS`的工作原理主要基于`SSL/TLS`握手过程。握手过程主要包括握手协商和密钥交换两部分。
握手协商部分，
首先客户端会发送一个`ClientHello`请求，这个请求包含`TLS`协议版本、一个随机数、一个密码套件列表；
服务器收到请求后，发送一个`ServerHello`响应，这个响应包含`TLS`协议版本、一个随机数、一个密码套件列表、服务器的数字证书
客户端收到`serverhello`响应后，会去验证数字证书是否由CA颁发，用CA的公钥解密得到服务器的公钥，客户端生成一个随机的对称加密密钥，并使用服务器的公钥加密这个秘钥，然后发送给服务器。
服务器使用自己的私钥解密得到客户端生成的对称加密秘钥，后续所有通信都使用对称密钥进行加密。
# TCP和UDP的区别
TCP和UDP都是传输层的协议。
TCP是**面向连接的协议，需要三次握手才能建立连接**，而UDP是无连接的，可以直接发送数据。
TCP提供可靠的数据传输，确保数据包的顺序和完整性；而UDP是不可靠的传输，不能保证顺序性和完整性
TCP具有拥塞控制和流量控制机制，可以根据网络状态来调整数据传输速率，通过滑动窗口进行流量控制，确保接收方能处理过来这些数据包；而UDP没有这些机制，发送速率固定。
TCP能够重传丢失的数据包，而UDP不提供错误恢复机制。
TCP的**报文头部**复杂，包含序列号、确认号等信息；而UDP的报文头部相对简单。
由于TCP需要建立连接，拥有**数据校验、重传等机制**，其**性能开销**通常比UDP大。
TCP适用于对可靠性要求高的场景，如网页浏览、文件传输；而UDP适用于对实时性要求高的应用，如语音通话、视频会议等。

| 特性         | TCP                                                    | UDP                          |
| ---------- | ------------------------------------------------------ | ---------------------------- |
| 是否面向连接     | **面向连接**（要先三次握手）                                       | **无连接**（直接发）                 |

| 数据传输方式     | 字节流（Stream）                                            | 数据报（Datagram，报文边界保留）         |
| 速度         | 较慢（握手、确认、流控、拥塞控制）                                      | 较快（没有这些开销）                   |

| 适用场景       | 要求 **可靠性** 的：网页、邮件、文件传输                                | 要求 **实时性** 的：视频、语音、直播、DNS、游戏 |
# TCP连接如何确保可靠性
TCP通过**差错控制、超时重传、流量控制、拥塞控制**等机制，确保了数据传输可靠性和效率。
首先是**差错控制**。接收方接受数据包会从TCP字节流中的**序列号**判断数据包的顺序是否正确。并通过**数据校验机制**，来判断数据包的内容是否正确。如果都正确的话，会向发送方返回一个ACK(`acknowledge`）确认信息。如果错误，接收方不会确认，触发超时发重传机制。
然后**超时重传机制**是**发送方**发送数据时会设置一个**定时器**，如果超时时间之前发送方没有收到确认信息，判断这个数据包丢失，重新传输这个数据包
流量控制，**接收方在ACK中通告一个窗口大小**给发送方，发送方通过**滑动窗口机制**调整发送数据包的数量，确保接收方能够处理发送方的数据量。
拥塞控制：TCP通过慢启动、拥塞避免、快重传和快恢复等算法，控制数据的发送速率，防止网络拥塞。

# 拥塞控制是怎么实现的？
TCP拥塞控制可以在网络出现拥塞时动态调整数据传输的速率，以防止网络过载。TCP拥塞控制的主要机制包括以下几个方面：
慢启动：TCP发送方会以一个较小的发送窗口开始传输数据，随着每次收到确认信息，发送方指数级增加发送窗口大小。这有助于在网络刚开始传输时谨慎地逐步增加速率，以避免引发拥塞。
拥塞避免：一旦达到慢启动设定的阈值，TCP发送方会进入拥塞避免阶段。发送方以线性级别增加发送窗口的大小。
快速重传：如果发送方连续收到相同的确认，它会认为数据包丢失，并快速重传未确认的数据包，而不必等待超时。这有助于更快地恢复由于拥塞引起的数据包丢失。
快速恢复：在发生快速重传后，TCP进入快速恢复阶段。发送方不会回到慢启动阶段，而是将慢启动阈值设置为当前窗口的一半，并将拥塞窗口大小设置为慢启动阈值+已确认但未被快速重传的数据块的数量。这有助于更快的从拥塞中恢复。

# TCP流量控制是怎么实现的？
流量控制是通过接收窗口和滑动窗口机制实现的。TCP接收方会在ACK确认报文中的窗口字段来告知发送方自己缓冲区的剩余空间，也就是接收窗口大小。发送方会动态调整发送速率，避免接收方缓冲区溢出。如果接收窗口缩小到0，发送方会暂停发送，并定期探测。只有等待接收方在ACK里更新接受窗口大小，发送方才会继续发送。这样避免了接收方缓存溢出，实现端到端的流量控制。
# UDP怎么实现可靠传输？
UDP是无连接的传输层协议，资源消耗小，处理速度快，但不能保证数据传输的可靠性。
在需要可靠传输的场景，可以参照TCP可靠性传输的方式，在应用层上添加超时重传和确认序列号等机制来实现可靠传输。具体来说：
在UDP发送方数据报中定义个首部，首部包含确认序列号和时间戳，时间戳是用来计算RTT，进而计算出合适的RTO（重传的超时时间）。然后当收到接收方的确认之后才发送下一个数据报；若超时未收到确认消息，发送方重新传输数据报。
UDP接收方接收到一个数据报之后，取下数据报首部，并添加本端的确认数据报首部，发送给对端。接收方根据序列号对已收到的数据进行排序并丢弃重复的数据报。这样就能在UDP之上构建一套可靠传输机制。

# TCP连接三次握手的过程，为什么是三次，可以是两次或者更多吗？
三次握手能够确认双方都具备收发能力，并且避免由于旧的连接请求导致的错误连接。
初始时，服务器通过`socket()`创建套接字，`bind()`绑定端口、`listen()`进入监听状态，等待客户端连接。客户端调用`connect()`发起连接请求。于是进入TCP建立连接的三次握手。
第一次握手。客户端向服务器发送一个`SYN(Synchronize)`报文，并随机选择一个初始序列号，请求建立连接。于是客户端进入`SYN_SENT`状态。
第二次握手。服务器收到`SYN`报文后，如果同意建立连接，则返回一个`SYN-ACK(Sychronize - Acknowledge)`响应报文，确认号为客户端发送的序列号+1，以及服务器自己也会初始化一个序列号，服务器进入`SYN_RCVD`状态。  
第三次握手。客户端收到服务器的`SYN-ACK`报文后，会发送一个`ACK`报文作为最终响应。确认号为第二次握手服务器初始化的序列号+1，之后客户端和服务器都进入`ESTABLISHED`状态，连接建立成功。
如果只有两次握手，会带来什么问题？考虑第一种情况，如果一个旧的`SYN`报文在网络延迟的情况下到达服务器，服务器会认为连接已经建立。然而客户端并没有真正的建立连接，会导致资源的浪费。第二种情况，当服务器回应一个`SYN-ACK`后，没有第三次握手的话，无法确认客户端能够收到这个响应。客户端可能因为丢包、网络延迟等没能建立连接。
三次握手已经能够确认双方的收发能力，又能防止历史报文引起的伪连接，所以不需要额外的握手开销。
# TCP连接四次挥手的过程，为什么是四次？
TCP断开连接时，客户端和服务器都可以主动发起。假设客户端先发起关闭请求：
第一次挥手，客户端发送一个`FIN(Finish)`报文给服务端，表示客户端断开数据传送，仍可接收数据。于是客户端进入`FIN-WAIT-1`状态。
第二次挥手，服务端收到`FIN`报文后，回复`ACK`报文给客户端。于是，服务端进入`CLOSE-WAIT`状态，客户端进入`FIN-WAIT-2`状态。
