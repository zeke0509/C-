# 一条SQL查询语句是如何执行的？
当客户端发送一条SQL给服务器时，服务端会经历几个阶段
1. 连接管理。客户端通过TCP连接到数据库服务器（操作系统层面），然后连接器会做**登录校验、权限管理、连接管理**。校验成功后，会分配一个线程，专门负责执行这个客户端的语句。
2. 查询缓存。这个机制在MySQL8.0后被取消了。MySQL拿到一个请求后，会先去查询缓存，如果缓存命中，就直接把缓存结果返回给客户端。
3. 如果缓存没有命中，会来到**分析器**中，对SQL语句进行**词法和语法分析**。词法分析主要去识别语句中的关键字，如`select、from、where`这些，语法分析是对识别出来的关键字进行分隔，判断当前的SQL指令是否合法。经过分析之后，分析器会构建一棵语法树，树中的节点是经过拆解之后的词法单位。
4. 接下来进入预处理阶段。进一步对语法树进行校验，检查表结构和字段的数据。如，检查被查询的表是否存在、\*会被扩展成表里对应的字段。
5. 下面是优化阶段，这一部分由优化器负责。优化器主要决定如何执行SQL才最优，比如用不用索引、JOIN用嵌套循环还是哈希连接，生成执行计划，将之前的语法树写入执行计划，演变成一个执行计划树
6. 执行器会根据优化器给的执行计划去调用**存储引擎**API完成读写操作，将取出满足条件的行，组装成结果集，通过MySQL协议把结果流式返回给客户端。
# 事务的四大特性有哪些
事务的四大特性，简称ACID
A`(Atomicity)`是指原子性：它表示事务中的操作要么全部执行成功，要么全部失败回滚，不存在部分成功的情况。举例来说，在交易的过程中，要么成功要么失败，不存在交钱未交货或者交货未交钱的情况。
C`(Consistency)`是指一致性，事务执行前后，数据库必须从一个一致的状态转变到另一个一致的状态。例如，转账前后，双方的总金额是一样的。
I`(Isolation)`是指隔离性。多个事务并发执行时，相互独立，一个事务的执行不影响其它事务
D`(Durability)`是指持久性。一旦事务被提交，他对数据库的改变是永久性的，不会因系统故障而丢失。

# 数据库的事务隔离级别有哪些？
数据库的事务隔离级别的提出主要是为了应对事务并发执行时出现的一些问题。这些问题有三个。
脏读：一个事务读到另一个事务修改过但未提交的数据。
不可重复读：一个事务在多次读取同一行数据结果不同。
幻读：同一事务中，读取行的数量或集合变了。
不同的隔离级别时为了规避这些现象，隔离级别越高，效率也越低。四个隔离级别从低到高介绍。
读未提交允许一个事务读取到另一个事务尚未提交的数据修改，这种情况下存在脏读、不可重复读和幻读的问题。
读提交。当一个事务提交之后，其它事务才能读到。解决了脏读问题，但仍可能出现不可重复读和幻读。
第三个隔离界别是可重复读，事务执行期间多次读取同一数据会得到相同的结果。解决了脏读、不可重复读问题，但仍可能出现幻读。
最高隔离级别是序列化，它会对读和写操作上锁。谁获取锁谁才能执行。一个事务对数据库进行操作，那么另一个事务必须等到当前事务执行结束并获取到锁才能执行操作。会解决脏读、不可重复读和幻读。但是由于把并行操作编程了串行，导致执行效率低。

# `MVCC`（多版本并发控制）无锁机制
在`undolog`中维护了三个的隐藏列：第一列表示产生该版本的事务ID；第二列表示旧数据的存储指针形成一条版本链；第三个ID只有当我们没有维护主机ID的时候会维护一个`row id`隐藏列。

这些历史数据存储在`undolog`中，那如何找到对应的历史数据呢？由`ReadView`定位实现。如果当前版本对`readview`不可见，就沿着`roll_pointer`回溯，直到中找到第一个可见版本。
以读已提交隔离级别为例，每一次查询都会创建一个`ReadView`。
`ReadView`中保存了**当前事务ID(`creator_trx_id`)、所有正在并发的事务ID集合（即未提交的事务），按照从小到大排列得到最小的未提交事务ID(min_trx_id)、未开始的事务(max_trx_id)**。可见性判断如下：
给定版本的事务id = 当前事务ID(`creator_trx_id`)，这是说明自己改的自己能看，所以可见。
给定版本的事务`id` < 正在并发事务的最小`id(min_trx_id`)，说明这条数据早于快照时刻已提交，所以可见。
若给定版本的事务`id` > 未开始事务的最大`id`，说明这条数据是快照之后才产生，所以不可见。
若给定版本的事务`id`在所有并发事务的`id`区间中，说明快照时刻仍为提交，所以不可见。
否则，介于`min`和`max`之间，但不在并发的事务`m_ids`中，说明当时已经提交，所以可见。


# MySQL的执行引擎有哪些？
`MySQL`的存储引擎主要有`InnoDB、MyISAM、Memory、Archive`，目前`MySQL`中默认用的存储引擎是`InnoDB`，它是这几个引擎中唯一提供事务ACID支持的。同时，它还有一些高级功能，如行级锁和外键。适用于需要事务和高并发的应用。但是它需要占用的数据空间会比较大，这是它的一个缺点。
`MyISAM`存储引擎，是早期的默认存储引擎。不支持事务，也不支持行级锁和外键约束，但是支持全文索引，访问速度快，适用于快速读取且读多写少的场景。
`Memory`是将数据都存储在内存中，访问速度快，但是服务器重启后数据会丢失。默认使用哈希索引，所以有一定局限性。如不同数据指向了一个哈希值，会造成哈希碰撞。适用于快速查找，但对数据丢失有一定容忍度，如临时表，缓存表。
`Archive`存储引擎，它支持高效插入，但是不支持索引，所以适用于归档日志。

# B树和B+树的插入删除操作
m阶B树要求：每个节点最多有m-1个关键字，m个孩子（分支）；除根结点外，最少有`(m/2 - 1)`个关键字，`m/2`个孩子（分支）（向上取整）
B树插入删除的核心就是：在保持中序遍历结果递增的同时，保证树的平衡和节点容量约束
## B树插入
- 先找到插入位置进行插入，插入位置一定会在叶节点
- 如果没有上溢出，无需调整
- 否则中间元素上移，两边分裂成两个节点。
## B树删除
- 删除非叶节点元素最终都转换成删除叶节点元素，替换为要删除节点的前驱或者后继
- 直接删除叶节点元素，并判断有没有下溢出。若没有下溢出无需调整
- 若有下溢出：左右兄弟够借，就直接借，即父节点下移到下溢出的节点，然后兄弟节点上去；左右兄弟都不够借，就执行合并操作，父节点下移到左，然后右侧合并过来，再检查父亲节点是否发生下溢。
